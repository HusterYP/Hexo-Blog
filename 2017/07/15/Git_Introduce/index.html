前言

一直以来对Git的概念比较模糊，近日从网上查看资料总结了一下，包括一些基本操作等

正文

什么是Git和版本控制

  • 简单的说Git是现在最先进的分布式版本控制系统
  • 比如说你在改一篇论文，要删除一个段落，但是你又怕改坏了，于是你就备份原文件，接着再改时又要备份，过了一周，你想找回被删除的文字，但是已经记不清删除前保存在哪个文件里了，只好一个一个文件去找，真麻烦。

  • Git就是帮助我们记录每次文件的改动的（版本控制）,如下

创建版本库

  • 版本库又名仓库，英文名repository，你可以简单理解成一个目录，这个目录里面的所有文件都可以被Git管理起来，每个文件的修改、删除，Git都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以“还原”。
  • mkdir可以创建目录（文件夹）
  • pwd显示路径
  • git init 把指定文件夹变成Git版本库，自动添加 .git目录，可以通过ls -ah查看

文件添加与提交

  • git add readme.txt 将文件添加到仓库
  • git commit -m "write something"将文件提交到仓库，""中的内容是本次提交的说明，可以做一些说明和提示，方便以后查看
  • cat fileName可以查看文件内容

后悔药

  • git status可以查看仓库当前状态，是否有文件被修改等
  • git diff可以查看修改内容
  • git log命令显示从最近到最远的提交日志，需要友情提示的是，你看到的一大串类似3628164...882e1e0的是commit id（版本号）;每提交一个新版本，实际上Git就会把它们自动串成一条时间线。如果使用可视化工具查看Git历史，就可以更清楚地看到提交历史的时间线；如果嫌输出信息太多，看得眼花缭乱的，可以试试加上--pretty=oneline参数,加上后，如：

其中的HEAD表示当前版本

  • git reset退回某个版本（自己在后面加参数）；git reset --hard ID表示退回哪一个版本，ID表示的版本号，注意这里的版本号不需要全写，只需要写前几位即可，Git会自动识别；但是此时虽然回到了旧版本，但是通过git log我们可以看到最新的版本已经不见了，那么要回到最新的版本怎么办喃，办法其实还是有的，只要上面的命令行窗口还没有被关掉，你就可以顺着往上找啊找啊，找到那个最新版本的commit id，于是就可以指定回到未来的某个版本
  • 但是若前一个ID不小心被清屏了怎么办，可以用git reflog来查看你的每一次命令

工作区与暂存区

  • 注意每次修改工作区的内容后要git add之后的git commit才能把修改提交到最新版本，因为git commit只是把暂存区中的修改提交了，而如果工作区的修改没有git add添加到暂存区的话是不会被提交的

撤销修改

  • git checkout --fileName可以撤销工作区修改；命令git checkout -- readme.txt意思就是，把readme.txt文件在工作区的修改全部撤销，这里有两种情况：

    • 一种是readme.txt自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；
    • 一种是readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。
    • 总之，就是让这个文件回到最近一次git commit或git add时的状态。
    • git checkout -- file命令中的--很重要，没有--，就变成了“切换到另一个分支”的命令,注意这里的--前后都有空格

  • git reset HEAD file可以把暂存区的修改撤销掉（unstage），重新放回工作区

  • 要注意区分用git status显示的状态哪种是还没有将修改提交到暂存区，即还未git add的，如下是还未git add

文件删除

  • 可以直接删除文件，或者在命令行中通过rm file也可以，但是这个只是删除了工作区中的文件，还没有删除版本库中的文件，也就是说此时通过git status或者git log还有其实区别，如果是不小心删错了，可以通过git checkout -- file将文件从版本库中恢复到工作区(注意--前后都有空格);若想要彻底删除文件，可以git rm file之后git commit -m "write something"更新版本库

关联GitHub与远程提交

  • 新建一个GitHub仓库，然后关联本地Git仓库，可以用git remote add origin git@github.com:HusterYP/GitControl.git命令，但是注意把HusterYP换成自己的GitHub用户名
  • 第一次推送master时，用git push -u origin master(其中origin是上面自己指定的名称);之后就不需要-u了

Git克隆GitHub仓库到本地

  • git clone git@github.com:HusterYP/fileName.git(其中fileName.git是GitHub上的仓库名称，注意把HusterYP换成自己的咯)

创建与合并分支

  • 分支的作用是为了不影响到主分支的内容，在自己的分支上工作，完成后再合并到主分支上
  • git checkout -b dev表示创建dev分支，并切换到dev分支（-b表示切换，相当于git branch dev和git checkout dev两条命令）
  • git branch可以列出所有分支，当前分支前面会有一个*号
  • 若分支完成工作后，主分支要合并该分支，那么可以先切换回主分支git checkout master,注意这时候访问在dev分支上的内容是不行的，因为还没有合并，git merge dev表示将分支dev与当前分支合并（注意是当前分支哈，不一定是主分支）
  • 之后可以删除dev分支了:git branch -d dev

分支策略

  • 首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活
  • 干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；你和你的小伙伴们每个人都在dev
    分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了
